## Task Details

### Task 1 -- Agent search filter

**BL coverage:** BL-206.

**File:** `components/RunDetail.astro`.  Line 82 has the search `<input type="text" placeholder="Search agents..." class="em-search" ...>`.  Currently decorative.

**Changes:**

1. Add a `<script>` tag at the bottom of `RunDetail.astro` (Astro auto-scopes scripts to the page).  No React island needed for this simple filter.

2. Script behavior:
   - On DOM ready, grab `document.querySelector('.em-search')` and agent card elements (likely `.agent-card` or the class used by `AgentCard.astro`).  Verify class name by reading `AgentCard.astro`.
   - Attach `input` event listener.
   - On input event: read query (lowercase, trim), iterate agent cards, set each card's `display` to `none` or default based on case-insensitive substring match against the agent's name.
   - Empty query: restore all cards.

3. Use a `data-agent-name` attribute on the AgentCard root element if not already present, rather than string-parsing the card DOM.  If `AgentCard.astro` doesn't have this attribute, add it (trivial pass).

4. Don't reinvent: no debounce needed for typical agent counts; can add later if 1000+ grid shows lag.

**Tests (T2):**

- `components/__tests__/RunDetail-search.test.ts` (or matching convention in the repo).
- Test cases:
  - Render with 3 agents, type "Al" in search, assert 2 cards hidden, 1 visible (for a seeded set where only "Alice" matches).
  - Empty search restores all cards.
  - No-match query hides all cards.
  - Case-insensitive match.
- Use Vitest + whatever DOM testing library emergence-ui already has (check existing tests for pattern).

**Acceptance criteria:**

- Filter works at 7 agents (small) and at 100 agents (simulated via test fixture; no need to load 1000 in tests).
- No console errors.
- Keyboard input works in all major browsers (Astro ships standard DOM).
- Tests pass.

### Task 2 -- Prev/next from actual day list

**BL coverage:** BL-207.

**File:** `components/DayDetail.astro`.  L17-18 currently:

```astro
const prevDay = day.sim_day > 0 ? day.sim_day - 1 : null;
const nextDay = day.sim_day < totalDays - 1 ? day.sim_day + 1 : null;
```

This arithmetic breaks when days are skipped (e.g., run exports days 0, 1, 3, 5 -- day 2 and 4 don't exist, so links to them 404).

**Changes:**

1. Upstream of DayDetail, the page builder (look at `pages/*.astro` or the page that consumes DayDetail) must pass an array of available day numbers for the run.

2. Add a prop to DayDetail: `dayList: number[]` (sorted ascending, integers).

3. Replace L17-18 with array-based lookup:
   ```astro
   const currentIndex = dayList.indexOf(day.sim_day);
   const prevDay = currentIndex > 0 ? dayList[currentIndex - 1] : null;
   const nextDay = currentIndex < dayList.length - 1 ? dayList[currentIndex + 1] : null;
   ```

4. Update every call site of DayDetail to pass `dayList`.  Day list source: whatever data layer call returns the run's days (check json-loader for `getDays(runId)` or derive from the run's days_exported + shard files).

5. If `dayList` is empty or current day not in list (data error), prev/next both null; optionally log a warning.

**Tests (T1 + T2):**

- T1: pure unit test of the prev/next derivation logic.  Extract the derivation to a helper if needed so it's easily testable.  Cases: contiguous, non-contiguous (skip day 2), first day, last day, single-day run.
- T2: render DayDetail with mocked dayList, assert rendered links point to correct day numbers.

**Acceptance criteria:**

- Non-contiguous day lists navigate correctly (no 404s).
- First day hides prev; last day hides next.
- Single-day run hides both.
- Tests cover each case.

**Deviation gate:** if `dayList` isn't easily derivable from existing data layer, BLOCK with options (add loader method vs. pass from page-builder vs. derive inline).
